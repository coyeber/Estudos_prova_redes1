Introdução às Redes

Seções 1.1 · 1.2 · 1.3 · 1.5

1.1 O que é a Internet? ▶

Visão de componentes

  • Hosts (sistemas finais): dispositivos conectados à internet (PCs, smartphones, TVs, sensores…). Executam aplicações.
  • Enlace de comunicação: fibra óptica, par trançado, rádio, satélite. Cada tipo tem sua taxa de transmissão (largura de banda) em bps.
  • Roteadores e comutadores de enlace: encaminham pacotes ao destino. Roteadores na rede núcleo; switches na borda.
  • ISPs (Provedores de Serviço de Internet): hierarquia — ISP de acesso → ISP regional → ISP Tier-1 (backbone). Interconectam-se via pontos de presença (PoPs), peering e IXPs.

Visão de serviços

  • A Internet como infraestrutura que provê serviços às aplicações distribuídas.
  • Oferece dois tipos de serviço via socket API: entrega confiável (TCP) e melhor esforço (UDP).

O que é um protocolo?

Define o formato e a ordem das mensagens enviadas/recebidas entre entidades e as ações tomadas em cada evento. Analogia: humanos combinam "Oi → Oi! → você tem hora?" antes de trocar informação.

1.2 A borda da rede ▶

Sistemas finais e redes de acesso

  • DSL: usa linha telefônica existente. Modem DSL → DSLAM na central → Internet. Assimétrico (download > upload).
  • Cabo coaxial (HFC): compartilhado entre vizinhos. Usa CMTS no headend. Também assimétrico.
  • FTTH: fibra direto ao domicílio. AON (ponto-a-ponto) ou PON (compartilhada).
  • Ethernet: acesso corporativo/universitário via par trançado a switches Ethernet.
  • Wi-Fi (802.11): ponto de acesso (AP) conectado a roteador doméstico, que conecta ao ISP de acesso.
  • Redes celulares (3G/4G/LTE): estações base → rede celular → Internet.

Meios físicos

  • Par trançado (UTP/STP): Cat 5/6, até ~10 Gbps em curtas distâncias.
  • Cabo coaxial: condutor central + blindagem. Alta largura de banda, compartilhado.
  • Fibra óptica: imune a EMI, baixíssima atenuação, até centenas de Gbps.
  • Rádio terrestre: Wi-Fi, 4G. Sofre com interferência e multicaminho.
  • Rádio via satélite: GEO (35.000 km, ~280 ms latência) vs LEO (Starlink).
1.3 O núcleo da rede — comutação ▶

Comutação de pacotes

Mensagem dividida em pacotes. Cada pacote usa a capacidade total do enlace e percorre roteadores com política store-and-forward.

Atraso store-and-forward (N enlaces, L bits, R bps):
d = N × (L / R)
  • Enfileiramento (queueing): pacotes aguardam no buffer se a taxa de chegada > taxa de saída. Pode causar perda se buffer cheio.
  • Tabela de encaminhamento: roteador consulta endereço destino → escolhe enlace de saída.

Comutação de circuitos

Recursos reservados fim-a-fim para toda a duração da chamada. Exemplos: rede telefônica tradicional.

  • FDM (Freq. Division Multiplexing): cada chamada ocupa uma faixa de frequência.
  • TDM (Time Division Multiplexing): cada chamada ocupa slots de tempo fixos.
  • Desvantagem: circuito fica ocioso quando não há dados sendo transmitidos.

Comutação de pacotes vs circuitos

  • Pacotes: melhor aproveitamento do enlace, mais simples, mas com variação de atraso (jitter).
  • Circuitos: garantia de recursos, previsível. Usado em telefonia tradicional.
  • A Internet usa comutação de pacotes — adequado para dados em rajada.

Estrutura da Internet (ISPs)

  • ISP de acesso → ISP regional → ISP Tier-1 (backbone global, ex: AT&T, NTT).
  • IXP (Internet Exchange Point): ponto neutro onde ISPs fazem peering sem pagar trânsito.
  • CDNs (ex: Google, Akamai): redes privadas próprias para levar conteúdo próximo ao usuário.
1.5 Atraso, perda, vazão e camadas de protocolo ▶

Tipos de atraso nodal

d_nodal = d_proc + d_fila + d_trans + d_prop
  • d_proc (processamento): verificar erros de bit, determinar enlace de saída. Tipicamente < µs.
  • d_fila (enfileiramento): espera para transmitir. Depende do tráfego. Mais variável.
  • d_trans (transmissão): L/R. Tempo para colocar todos os bits no enlace.
  • d_prop (propagação): d/s, onde d = distância, s ≈ 2×10⁸ m/s. Independe de L.

Intensidade de tráfego e perda

I = La/R    (a = taxa média de chegada de pacotes)

I → 0 : atraso pequeno I → 1 : atraso → ∞ I > 1 : perda de pacotes

Traceroute

Ferramenta de diagnóstico que envia pacotes com TTL crescente (1, 2, 3…) para descobrir os roteadores no caminho e medir RTTs.

Vazão (Throughput)

  • Instântanea: taxa em um instante.
  • Média: F bits / T segundos.
  • Limitada pelo gargalo (enlace de menor capacidade no caminho).

Camadas de protocolo

  • Aplicação: HTTP, FTP, SMTP, DNS. Mensagens.
  • Transporte: TCP, UDP. Segmentos.
  • Rede: IP, roteamento. Datagramas.
  • Enlace: Ethernet, Wi-Fi. Quadros.
  • Física: bits no meio físico.

Cada camada encapsula a PDU da camada superior adicionando seu cabeçalho (encapsulamento).

⚠ Conceito importante: camadas fornecem serviços à camada superior e usam serviços da camada inferior. Esse isolamento permite evolução independente de cada camada.

Camada de Aplicação

2.1 Princípios das aplicações de rede ▶

Arquiteturas de aplicação

  • Cliente-servidor: servidor sempre ativo, IP fixo/estático, muitos clientes se comunicam com ele. Ex: HTTP, FTP, SMTP.
  • P2P pura: sem servidor sempre ativo. Pares se comunicam diretamente. Auto-escalável. Difícil de gerenciar. Ex: BitTorrent.
  • Híbrida: combina C/S e P2P. Ex: Skype (servidor para localização, P2P para voz).

Comunicação entre processos

  • Processo = programa em execução. Processos em hosts diferentes se comunicam por sockets.
  • Socket = interface entre camada de aplicação e de transporte (API).
  • Endereçamento: endereço IP (host) + número de porta (processo). Ex: 80 (HTTP), 443 (HTTPS), 25 (SMTP).

Serviços que uma aplicação pode exigir

  • Transferência confiável de dados: garantia de entrega (TCP) vs melhor esforço (UDP).
  • Vazão: apps sensíveis à largura de banda (streaming) vs elásticas (e-mail).
  • Temporização: requisitos de atraso máximo (jogos, VoIP).
  • Segurança: criptografia, autenticação (TLS sobre TCP).

TCP vs UDP — visão da aplicação

  • TCP: orientado a conexão (handshake), confiável, controle de fluxo e congestionamento. Sem garantia de temporização/vazão.
  • UDP: sem conexão, sem confiabilidade, sem controle de congestionamento. Simples e rápido. Usado em streaming e DNS.
2.2 A Web e o HTTP ▶

Visão geral do HTTP

  • Protocolo da camada de aplicação da Web. Usa TCP (porta 80 / 443 com TLS).
  • Stateless: servidor não mantém estado sobre os clientes.
  • Objetos endereçados por URL: protocolo + hostname + caminho.

Conexões HTTP

  • Não-persistente: nova conexão TCP para cada objeto. RTT extra para cada objeto. d = 2RTT + tempo_transmissão por objeto.
  • Persistente (padrão HTTP/1.1): múltiplos objetos na mesma conexão. Sem pipeline ou com pipeline (envia sem esperar resposta).
RTT = tempo para pacote ir do cliente ao servidor e retornar
d_resposta (não-persistente) = 2 RTT + T_trans(objeto HTML)

Mensagens HTTP

  • Request: linha de requisição (método URL versão) + cabeçalhos + corpo.
    Métodos: GET, POST, HEAD, PUT, DELETE.
  • Response: linha de status (versão código frase) + cabeçalhos + corpo.
    Códigos: 200 OK, 301 Moved Permanently, 304 Not Modified, 400 Bad Request, 404 Not Found, 505 Version Not Supported.

Cookies

  • Mecanismo para adicionar estado ao HTTP stateless.
  • 4 componentes: cabeçalho Set-Cookie na resposta, cabeçalho Cookie na requisição, arquivo no navegador, base de dados no servidor.
  • Usos: autenticação, carrinhos de compra, rastreamento de usuário.

Web caches (proxies)

  • Armazena cópias de objetos recentemente requisitados. Atua como servidor para clientes e como cliente para o servidor original.
  • Reduz tempo de resposta e tráfego no enlace de acesso.
  • GET condicional: proxy usa If-Modified-Since; servidor responde 304 se não houve modificação, economizando transmissão do objeto.
2.5 DNS — Domain Name System ▶

O que o DNS faz

  • Traduz nomes de domínio legíveis (www.google.com) em endereços IP (142.250.x.x).
  • Outros serviços: apelidos de host (alias), apelidos de servidor de e-mail, distribuição de carga (múltiplos IPs para um nome).
  • Base de dados distribuída e hierárquica, implementada em servidores de nome ao redor do mundo.

Hierarquia de servidores DNS

  • Root DNS: 13 clusters lógicos (a–m.root-servers.net). Conhecem os servidores TLD.
  • TLD (Top-Level Domain): .com, .org, .net, .edu, .br… Mantidos por organizações (Verisign, NIC.br).
  • Autoritativo: mantido pela própria organização ou ISP. Tem os registros oficiais do domínio.
  • DNS local (resolver): no ISP ou organização. Recebe consultas dos hosts; faz as perguntas em nome do cliente.

Resolução de nomes

  • Iterativa: servidor local pergunta a cada servidor, recebe referência ao próximo. Faz o trabalho o servidor local.
  • Recursiva: cada servidor pergunta ao próximo e retorna a resposta ao solicitante anterior. Mais carga nos servidores superiores.

Cache DNS

Respostas são armazenadas em cache com TTL. Servidores TLD frequentemente ficam em cache no servidor local, evitando consultas à raiz.

Registros DNS (RRs)

Formato: (Name, Value, Type, TTL)

Type A : Name = hostname, Value = IP Type NS : Name = domínio, Value = servidor DNS autoritativo Type CNAME: Name = alias, Value = nome canônico Type MX : Name = domínio, Value = servidor de e-mail

Protocolo DNS

  • Usa UDP, porta 53 (TCP para respostas > 512 bytes ou transferência de zona).
  • Mensagem de consulta e resposta têm o mesmo formato (flag QR distingue).
  • Campos: identificação, flags, perguntas, respostas, autoridade, informações adicionais.
2.7 Programação de sockets (conceitos) ▶
⚠ A programação em si (código) não é cobrada. Foco nos conceitos de socket e no modelo de comunicação.

O que é um socket

Interface (porta) entre o processo da aplicação e o protocolo de transporte. É a API de rede. O desenvolvedor tem controle de tudo na aplicação e do lado do transporte; o restante é gerenciado pelo SO.

Sockets UDP

  • Sem conexão prévia. Cada pacote carrega endereço IP destino e porta.
  • Não confiável, pode chegar fora de ordem ou ser perdido.
  • Fluxo: cria socket → sendto (endereço destino + dados) → recvfrom (recebe + endereço origem).

Sockets TCP

  • Orientado a conexão: 3-way handshake antes de trocar dados.
  • Servidor: cria socket de escuta (welcomeSocket) → accept() → cria novo socket de conexão para aquele cliente.
  • Cliente: cria socket → connect() (dispara handshake) → send/recv.
  • Cada par (cliente ↔ servidor) tem seu próprio socket dedicado.

Camada de Transporte

Seções 3.1 · 3.2 · 3.3 · 3.5 (exceto 3.5.3 e 3.5.5)

3.1 Serviços da camada de transporte ▶

Função principal

Provê comunicação lógica entre processos em hosts diferentes. A camada de rede provê comunicação entre hosts; a de transporte estende isso até os processos.

Relação com a camada de rede

  • Camada de rede (IP): serviço de melhor esforço, sem garantias. Pode perder, duplicar, reordenar datagramas.
  • Camada de transporte pode: adicionar confiabilidade (TCP), prover multiplexação, controle de congestionamento.
  • O que a camada de transporte não pode melhorar: largura de banda, atraso de propagação — esses dependem da rede.

TCP vs UDP — resumo

  • TCP: confiável, ordenado, controle de fluxo, controle de congestionamento, orientado a conexão. Segmentos.
  • UDP: não confiável, sem ordem, sem controle. Datagramas. Menor sobrecarga.
segmento datagrama UDP PDU encapsulamento
3.2 Multiplexação e demultiplexação ▶

Definições

  • Multiplexação (no emissor): coletar dados de vários sockets, encapsular com cabeçalho de transporte e passar à camada de rede.
  • Demultiplexação (no receptor): examinar campos do segmento para identificar o socket correto e entregar os dados.

Como funciona a demultiplexação

Segmento tem campos: IP fonte, IP destino, porta fonte, porta destino. O receptor usa esses campos para direcionar ao socket certo.

Sem conexão (UDP)

  • Socket UDP identificado por: (IP destino, porta destino).
  • Dois segmentos com IPs de origem diferentes, mas mesmo IP/porta destino → vão para o mesmo socket.

Orientado a conexão (TCP)

  • Socket TCP identificado por: (IP fonte, porta fonte, IP destino, porta destino) — 4-tupla.
  • Dois segmentos com IP de origem diferente, mas mesma porta destino → sockets diferentes.
  • Servidor com múltiplos clientes: um socket por conexão ativa.
3.3 UDP — Transporte sem conexão ▶

Por que usar UDP?

  • Sem handshake → menor atraso de estabelecimento.
  • Sem estado de conexão no sender/receiver → servidor suporta mais clientes.
  • Menor cabeçalho (8 bytes vs ≥ 20 bytes TCP).
  • Sem controle de congestionamento → pode enviar na máxima taxa suportada.
  • Aplicações: DNS, SNMP, streaming multimídia, jogos online, QUIC.

Estrutura do segmento UDP

| Porta fonte (16 bits) | Porta destino (16 bits) | | Comprimento (16 bits) | Checksum (16 bits) | | Dados (payload) |
  • Comprimento: tamanho total em bytes (cabeçalho + dados). Mínimo = 8.
  • Checksum: detecta (não corrige) erros de bit. Calculado sobre pseudo-cabeçalho + cabeçalho + dados.

Checksum UDP

  • Soma em complemento de 1 das palavras de 16 bits.
  • Emissor: calcula, coloca no campo checksum (complemento do resultado).
  • Receptor: soma tudo incluindo checksum → deve ser 1111...1 (todos 1s) se sem erro.
  • Se checksum = 0 (não habilitado em IPv4, opcional): campo zerado.
⚠ UDP oferece apenas detecção de erro (checksum), não correção nem retransmissão. A aplicação é responsável por qualquer confiabilidade adicional.
3.5 TCP — Transporte orientado a conexão ▶
📌 Subtópicos 3.5.3 (estimativa de RTT e timeout) e 3.5.5 (controle de fluxo) não são cobrados.

Características do TCP (3.5.1)

  • Ponto-a-ponto: um emissor, um receptor.
  • Fluxo de bytes confiável e ordenado: sem fronteiras de mensagem ("byte stream").
  • Full-duplex: dados fluem nos dois sentidos na mesma conexão. MSS = Maximum Segment Size.
  • Handshake: troca de variáveis de controle antes de transmitir dados.
  • Buffers de envio e recepção: TCP faz o melhor para entregar os dados, sujeto ao controle de fluxo e congestionamento.

Estrutura do segmento TCP (3.5.2)

| Porta fonte (16) | Porta destino (16) | | Número de sequência (32 bits) | | Número de ACK (32 bits) | | Len(4)|Reserv.|Flags(6)| Janela (16) | | Checksum (16) | Urgente (16) | | Opções (variável) | | Dados |
  • Número de sequência: byte offset do primeiro byte de dados do segmento no fluxo.
  • Número de ACK: próximo byte esperado do outro lado. ACK cumulativo.
  • Flags: SYN, ACK, FIN, RST, PSH, URG.
  • Janela de recepção (rwnd): usada para controle de fluxo.

Transferência confiável de dados (3.5.4)

  • TCP cria serviço confiável sobre o IP (não confiável).
  • Temporizador de retransmissão: disparado ao enviar segmento. Se ACK não chega antes do timeout → retransmite.
  • ACK cumulativo: confirma todos os bytes até o número de ACK.
  • Retransmissão rápida: ao receber 3 ACKs duplicados → retransmite segmento perdido sem esperar timeout. (Indica perda, não congestionamento severo.)
  • Cenários de retransmissão: segmento perdido, ACK perdido, ACK prematuro.
ACK duplicado: receptor recebeu segmento fora de ordem → reenvia ACK do último byte em ordem recebido.

3 ACKs dup → retransmissão rápida (fast retransmit)

Gerenciamento de conexão TCP (3.5.6)

  • Estabelecimento — 3-way handshake:
    1. Cliente → SYN (seq=x)
    2. Servidor → SYN+ACK (seq=y, ack=x+1)
    3. Cliente → ACK (seq=x+1, ack=y+1) + pode conter dados
  • Encerramento — 4 segmentos (FIN/ACK):
    1. Cliente → FIN
    2. Servidor → ACK
    3. Servidor → FIN
    4. Cliente → ACK (aguarda 2×MSL antes de fechar — TIME_WAIT)
  • Half-close: um lado pode fechar seu sentido enquanto ainda recebe dados do outro.
  • RST: abortar conexão imediatamente (sem FIN).
⚠ O estado TIME_WAIT garante que o ACK final chegue ao servidor e que segmentos "zumbis" da conexão expirem antes de uma nova conexão reutilizar os mesmos endereços e portas.
